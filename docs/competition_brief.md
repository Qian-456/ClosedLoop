# 比赛项目与评分规则（通用要求与赛道专属说明）

## 一、通用评审维度（大赛整体要求）

### 1. 创新性

- 解决方案切入角度是否新颖。
- 是否提出突破性的思路。
- 是否将 AI 技术与命题需求进行了创造性结合。

### 2. 完整性

- 搭建完整度及其与业务的集成程度。
- 功能设计的逻辑性、用户体验的友好度。
- 技术实现的成熟度与整体方案的稳定性。

### 3. 应用效果

- 用户体验质量：交互自然度、响应及时性、准确性。
- 需结合真实数据进行训练与优化，确保准确率满足真实用户需求。

### 4. 商业价值

- 市场潜力。
- 对品牌影响力与用户忠诚度提升。
- 是否能为本地生活业务带来明显经济效益或竞争优势，具备长期价值。

## 二、赛道专属描述：帮用户「把事情做完」

### 1. 任务背景

- **解决周末短时（4-6小时）出游痛点**。
- **场景覆盖**：“亲子（带娃/健身限制）”与“朋友（多人聚会）”出游。
- **核心诉求**：不仅是推荐，更要查座位、排队并规划“玩→吃→后续”完整链路。

*示例场景：周六上午 9 点，小明迎来了难得的周末双休，他给平台发了一条消息：“今天下午是空的，想和老婆孩子/朋友出去玩几个小时，别离家太远，帮我安排一下。”（亲子场景：孩子5岁，老婆最近在减肥；朋友场景：总共4个人，2男2女）。然后他把手机递给了老婆 / 发给了朋友，看看这个安排好不好。*

注意：这不是简单的搜索推荐，而是要在此基础上**帮你把事情做完**。平台需要在接下来的几分钟里：

- 规划出一条「下午 4–6 小时」的综合方案，例如：去哪玩 → 玩完去哪吃 → 有没有额外活动。
- 查到是否有适合不同群体要求的餐厅，以及餐厅有没有位置、是否需要提前排队（Mock）。
- 安排吃饭前后的活动，例如亲子乐园、展览、citywalk、小吃街等。
- 将方案转为可执行、可落地的动作：用户确认后可一键安排下单/预约/排队/通知/增值服务等（Mock）。
- 支持把计划发给他人，例如：“搞定了，下午 2 点出发，先去……”（Mock）。

### 2. 任务定义

构建一个本地场景 Agent，接受一句自然语言目标，输出可执行的综合方案，并自动完成下单、预订等关键动作。

## 三、赛道专属评分规则与硬性约束

### 1. 完整性（专属要求）

- **规划质量**：准确理解并满足人群、地理等约束；时间分配自然且不超过窗口；方案覆盖多环节完整链路。
- **执行质量**：标准场景工具调用的精确/召回率高；端到端跑通成功率（Pass\@1）高；具备自动异常处理能力。

### 2. 创新性（专属要求）

- **技术创新**：独特的多约束拆解思路（如动态时间分配）；并行的工具链路协作模式与智能容错。
- **交互体验**：方案确认直观可视化；确认后一键顺滑执行；支持多用户分享、协同确认及反馈闭环。

### 3. 加分项与约束条件

- **加分项**：代码结构模块化；一键部署简便；方案可行性高（优选 Mock API）。
- **速度约束**：方案生成 ≤ 30秒；工具响应 ≤ 3秒；端到端流程 ≤ 2分钟。
- **异常覆盖**：至少处理3类故障(无座/无票/冲突)。
