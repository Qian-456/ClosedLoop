# 会话搜索引擎与 Agent 缓存设计

## 概述

本设计包含两个联动目标：

1. 将当前 `search_candidates` 从默认依赖 session 级语义检索的模式，调整为更接近本地生活平台搜索推荐引擎的“关键词/标签/规则排序优先”模式。
2. 将当前 `build_agent()` 的重复初始化模式，调整为“进程内模型客户端与 Agent 工厂复用”模式，降低每次创建 Agent 的固定时间成本。

本设计不追求一次性推翻现有能力，而是优先降低延时、减少超时、提升结果可解释性，并保留后续按需恢复语义增强的空间。

## 背景与问题

### 当前搜索问题

- 当前会话内搜索的数据范围，本质上是单次规划后的 session 候选池，而不是全量开放 POI 库。
- 在这样的小候选池上继续维持 `Milvus collection + embedding + hybrid search + session 索引生命周期管理`，系统成本偏高。
- 搜索慢和超时的主要来源，不完全是文本匹配本身，而是：
  - session collection 的构建与加载
  - embedding 兜底
  - hybrid search 链路
  - 远程调用与索引状态不稳定
- 当前代码其实已经存在关键词兜底搜索，但它只是失败分支，不是主路径。

### 当前模型初始化问题

- `build_agent()` 每次调用都会重新：
  - 创建 `ChatDeepSeek`
  - 创建 `ChatTongyi`
  - 组装 middleware
  - 调用 `create_agent(...)`
- 当前主要使用远端模型 API，因此真正昂贵的并不是“本地模型权重装载”，而是反复创建客户端与 Agent 对象的固定初始化成本。

## 设计目标

- 将会话内搜索的默认路径改为低延时、低波动、强可解释。
- 将会话内搜索的结果逻辑做成“Mock 美团搜索推荐引擎”风格，而不是通用 RAG 风格。
- 保持与现有 `constraints`、`rerank score`、结构化 POI 字段契约兼容。
- 将模型与 Agent 的初始化改成进程内复用，减少重复构建。
- 保持上层调用接口不变，尽量不破坏 graph、tool 和 core 层的现有契约。

## 非目标

- 本次不设计大规模全量 POI 搜索系统。
- 本次不以语义搜索质量最大化为目标。
- 本次不直接删除所有旧的向量检索代码。
- 本次不引入独立的模型服务 daemon，也不引入本地大模型推理服务。

## 方案选择

### 已选方向

- 搜索采用：`方法 A`
  - 关键词/标签/规则排序优先
  - 默认关闭语义检索
  - 语义增强只保留为未来可开关能力
- 模型复用采用：`方案 1`
  - 进程内复用模型客户端
  - 进程内复用 Agent 工厂

### 为什么不选 RAG-first

- 当前 session 候选池规模较小，不值得为其维持高成本索引系统。
- 搜索结果需要可解释，平台检索式规则比向量分数更容易解释。
- 目前用户最看重的是低延时稳定，而不是更强的抽象语义理解。

## 搜索引擎设计

### 总体思路

`search_candidates` 主链路改为：

1. Query 归一化
2. 字段匹配召回
3. 结构化约束重排
4. 结果解释输出

默认不走 session 向量检索。

### 数据范围

- 搜索范围严格限定为当前 `session_id` 下已有候选池。
- 候选池继续来自当前规划结果的上下文，不扩展为跨 session 或全库开放检索。
- 数据源继续复用当前缓存的候选数据，不依赖新建 session collection。

### Query 归一化

将用户搜索语句拆成以下信息：

- 关键词：如“便宜”“儿童设施”“安静”“清淡”“适合亲子”
- 排序偏好：如“便宜点”“近一点”“评分高一点”
- 否定词：如“不要太吵”“别太贵”“不要礼物”
- 结构化偏好：映射到已有契约字段

归一化后生成内部查询结构，例如：

- 文本关键词列表
- 属性偏好列表
- 排序权重意图
- 否定条件列表

### 字段匹配召回

召回不依赖 embedding，而是直接对 session 候选池做字段匹配。

优先匹配字段包括：

- 文本字段：
  - `name`
  - `features`
  - `description`
  - `tags`
  - `experience_tag`
- 结构化字段：
  - `suitable_groups`
  - `child_facility_tags`
  - `kid_menu_status`
  - `stroller_friendly_status`
  - `gift_type`

### 排序逻辑

结果排序由显式规则分构成，建议按以下维度加权：

- 文本命中分
  - 关键词命中的字段数量
  - 命中字段的重要程度
- 结构化命中分
  - 是否命中儿童设施
  - 是否命中亲子友好字段
  - 是否命中 suitable_groups
- 历史约束一致性分
  - 当前 `constraints` 中的预算、距离、亲子、饮食等偏好是否匹配
- 原始候选质量分
  - 复用已有 `rerank score`
- 业务排序分
  - 价格更低
  - 距离更近
  - 亲子友好程度更高
  - 噪音更低

### 结果解释

每个返回项除了基础字段，还要附带简洁的命中原因，例如：

- 命中儿童设施
- 命中亲子标签
- 价格更低
- 更符合当前预算
- 更适合安静场景

这样 Agent 在向用户解释“为什么推荐这些结果”时，不再需要猜测向量相似度来源。

### 语义增强策略

- 默认关闭
- 不进入主路径
- 后续若启用，只作为实验开关挂在“结果过少/命中分过低”的兜底分支上
- 不影响默认稳定路径

## Agent 与模型缓存设计

### 总体思路

将 `build_agent()` 拆成两层复用：

1. 模型客户端缓存
2. Agent 工厂缓存

### 模型客户端缓存

缓存对象包括：

- `ChatDeepSeek`
- `ChatTongyi`

建议缓存 key 由以下组合决定：

- provider
- model
- temperature
- timeout

这样可以确保：

- 相同配置复用同一个客户端
- 配置变化时自动创建新实例

### Agent 工厂缓存

对固定结构的 Agent 构造进行缓存，例如：

- `plan_agent`
- `execute_agent`
- 常见结构化输出 Agent

缓存 key 应由以下维度决定：

- tools 签名
- middleware 签名
- state_schema
- response_format
- checkpointer 是否存在及其类型特征

### 缓存边界

允许缓存：

- 模型客户端
- 固定配置 Agent
- 结构化输出包装器

禁止缓存：

- 某次调用的 `messages`
- 某个用户会话的运行态状态
- `thread_id`
- 具体的请求输入

目标是复用初始化成本，而不是共享会话状态。

### 失败与降级策略

- 缓存 miss：正常初始化并回填缓存
- 缓存对象异常：剔除异常实例并重建
- 不允许损坏对象长期滞留在缓存中

## 兼容性与迁移策略

### 搜索迁移

第一阶段：

- `search_candidates` 默认切到关键词/规则检索主路径
- 旧语义搜索代码保留，但不作为默认路径

第二阶段：

- 根据真实稳定性与效果数据，决定是否继续保留 session 级向量检索

### 模型迁移

- 对上层调用保持 `build_agent(...)` 不变
- `closedloop.graph.agent`、graph nodes、tools 无需改动调用姿势
- 缓存逻辑封装在 `core.llm` 内部

## 测试策略

### 搜索测试

必须覆盖：

- 文本字段命中
- 结构化字段命中
- `constraints` 继承重排
- 否定词处理
- 空候选池返回
- 命中原因输出

### 模型缓存测试

必须覆盖：

- 相同配置命中缓存
- 不同配置正确分流
- 异常实例可被重建
- 结构化输出与普通 Agent 均能复用

## 验收标准

### 搜索侧

- 会话内搜索平均耗时显著下降
- 超时率、连接失败率显著下降
- 结果解释更直接，适合 Agent 向用户说明

### 模型侧

- 重复调用 `build_agent()` 的固定初始化开销明显下降
- 不破坏现有 graph/tool 契约
- 不引入会话状态串扰

## 风险

- 关键词/规则搜索对极其抽象的自然语言 query 可能不如语义检索灵活。
- 若排序规则设计不当，可能会出现“命中了词但排序不自然”的问题。
- Agent 工厂缓存若边界设计错误，可能带来隐性共享状态风险。

## 最终建议

先落地：

1. 平台检索引擎式的会话搜索主路径
2. `build_agent()` 的进程内缓存复用

暂不优先投入：

1. session 级 Milvus 搜索性能继续打磨
2. 默认语义增强链路
3. 独立模型服务持久化

这样最符合当前项目“低延时稳定优先”的目标，也最贴合当前 Mock 本地生活数据与 Agent 工作流。
