# Plan Trip 功能与排队预估机制整合设计 (Design Spec)

## 1. 目标与背景

为了让 `plan_trip` 规划功能更符合真实世界的物理与工程逻辑，同时极大地增强 Agent 在用户面前的“智能感”与“贴心感”，本设计明确了**预估排队时间（Wait Time Estimation）**的展示策略与 Agent 的介入博弈机制。

核心原则：
1. **切断上帝视角**：规划阶段不再读取真实的 Mock 数据库占用情况，而是基于店铺的热度（Rating 等）进行经验性预估。
2. **二次函数惩罚**：对高无座风险（`seating_risk_prob`）的备选方案施加 `(prob ^ 2) * 50` 的非线性惩罚，精准规避高风险选项。
3. **弹性静默与动态博弈结合**：在不打破硬约束的前提下静默替换；在打破约束时，Agent 提供协商方案。

## 2. 核心设计：打造“有血有肉”的智能规划师

将系统的数据支撑（Engineering）与 Agent 的拟人化话术（Workflow/UX）深度结合。

### 2.1 底层数据支撑（已实现）
后端的 `ItineraryItem` 结构中已经包含了 `duration_breakdown`，其结构如下：
```json
"duration_breakdown": {
    "base_minutes": 60,
    "wait_minutes": 30,  // 预估排队时间
    "buffer_minutes": 5,
    "total_minutes": 95
}
```
这是前端 UI 拆解和 Agent 话术生成的物质基础。

### 2.2 场景 A：首次生成完美行程（话术邀功 + UI 透明）
- **触发时机**：用户首次发起规划请求，系统成功返回 `itinerary`。
- **Agent 行为 (Prompt 约束)**：Agent 必须观察 `itinerary` 中的数据。如果发现有热点店铺（`wait_minutes > 0`），Agent 必须在生成的摘要中主动“邀功”。
  - *话术示例*：“我为您规划好了行程。注意到您选的『海底捞』是 4.8 分的热门餐厅，**我凭借经验为您额外预留了 30 分钟的等位时间**，确保行程不赶。请看下方的方案。”
- **前端 UI 展现 (契约)**：在“推荐方案”卡片的时间轴节点上，前端应将纯耗时与预估排队时间拆解显示。例如打上显眼的标签：`[🍽️ 用餐 60分钟] [🔥 预估等位 30分钟]`。

### 2.3 场景 B：用户发起替换，且未打破硬约束（静默替换 + 话术邀功）
- **触发时机**：用户要求换一家店，系统在 `backup_candidates` 或通过搜索找到替换项，且替换后的总时间/预算仍在限制内。
- **Agent 行为**：后台直接完成替换。Agent 在回复时，点出排队时间的变化。
  - *话术示例*：“没问题，已经为您静默替换为『八合里牛肉火锅』。不过这家店比较火，**我把预估等位时间上调到了 45 分钟**，好在您原本的时间比较充裕，整体行程依然顺畅！”

### 2.4 场景 C：用户发起替换，且打破了硬约束（动态博弈协商）
- **触发时机**：用户要求换的热门店导致预估排队时间大幅增加，进而触发了 `validator` 的超时报错（硬约束打破）。
- **Agent 行为 (Conflict Resolver)**：Agent 不应直接拒绝，而是抛出带有选项的博弈回复，体现“协商感”。
  - *话术示例*：“这家店确实不错，但它的预估等位长达 45 分钟，如果换成它，**会导致您的整个行程超时**。您看这样好不好：\n1. 我们坚持去这家，但我帮您把后面的『下午茶』删掉腾出时间？\n2. 我帮您找一家口味相近，但评分 4.0 左右、**基本不用排队**的平替餐厅？”

## 3. 落地实施要求

为了将本设计完全落地，需要在后续的开发阶段跟进以下改动：

1. **前端适配**：前端组件需要适配 `duration_breakdown`，渲染排队高亮标签。（遵循 Frontend Contract）
2. **Agent Prompt 升级**：修改 `backend/src/closedloop/graph/prompts/` 下的系统提示词（特别是总结/规划后置节点），强制要求 Agent 根据 `wait_minutes` 进行拟人化的邀功表达。
3. **Validator 异常信息丰富化**：当产生冲突时，底层的验证器返回给 Agent 的错误信息中，需要明确指出是因为“排队时间过长”导致的超时，以便 Agent 能够顺畅地生成上述的“博弈选项”。
