# Demo 展示与复现（ClosedLoop 执行型本地生活 Agent）

## 一句话目标

用户说一句话 → 系统生成 4–6 小时“可执行行程”（玩→吃→后续）→ 用户确认 → 自动执行预约/排队/通知（Mock）→ 关键失败场景可自动补救（例如餐厅无座自动替换）。

## Demo 环境准备（评委友好版）

本项目所有执行均为 Mock：不会进行真实下单、真实网络调用或真实扣款。

你可以任选一种方式运行：

- 本地运行（推荐开发者）：启动主服务 + 规划子服务（端口分别为 8000/8001）。
- Docker 运行（推荐评委）：一键启动全部服务后再进行 Demo。

如果你只看演示效果：Web UI 或命令行均可完成“输入→出方案→确认→执行→故障→自愈”的完整链路。

## 90 秒主路径脚本（朋友聚会版）

> 目标：让评委一眼看到“把事情做完”的差异点：执行中遇到无座不报错、先自愈。

1) 输入：

> 今天下午 14:00 出发，4 个人（2男2女）聚会，想玩 4–6 小时，别太远，安排“玩→吃→后续”。

2) 系统输出：

- 一条 4–6 小时行程（含：活动 → 晚餐 → 饭后轻量活动/散步）
- 每个步骤包含：时间段、地点、预计耗时、（可能的）预约/排队提示
- 仅在聊天区给摘要，并提示“详细方案在推荐面板/方案卡片”

3) 用户确认：

> 就按这个方案执行（或“帮我把都预约了”）

4) 执行过程可见：

- 系统逐步展示：正在预约/正在取号/正在通知…
- 关键展示点：餐厅预订失败（无座）后，系统自动切换同槽位备选餐厅并继续执行

5) 完成：

- 输出执行结果摘要：哪些预约成功、哪些替换过、最终集合时间/路线摘要
- 可选：生成一段可复制的“分享文本”

## 3 分钟复现步骤（含故障注入）

### Step 1：确保“无座”一定触发

在 Mock 数据里，将某个“晚餐套餐”在目标时段的 `capacity_remaining` 改成 `0`，即可稳定触发无座。

文件：`mock_db/reservations.json`（或服务实际读取的同名文件）

示例（伪示例，字段与含义与现有结构一致）：

```json
{
  "target_type": "combo",
  "target_id": "combo_xxx",
  "time_slots": [
    {
      "start_time": "17:00",
      "end_time": "19:00",
      "capacity_total": 30,
      "capacity_remaining": 0,
      "queue_required": true,
      "wait_minutes": 90
    }
  ]
}
```

说明：

- `capacity_remaining = 0` 表示满员/无座
- `queue_required / wait_minutes` 用于拟真排队（可选展示）

### Step 2：发起规划请求（得到方案）

发送自然语言需求，得到包含餐厅步骤的方案。

观察点：

- 方案是 4–6 小时窗口内的完整链路（玩→吃→后续）
- 餐厅步骤存在“可执行属性”（可预约/可排队的提示）

### Step 3：确认执行（触发 Mock 执行器）

用自然语言确认（例如“就这个方案，帮我预订”），系统进入执行阶段。

观察点：

- 执行过程逐步可见（步骤级状态）
- 当遇到无座时，不要求用户立刻做复杂选择：系统优先尝试“同槽位备选替换”

### Step 4：验证自愈成功

观察点（至少满足其中 2 项即可认为演示成功）：

- 看到“无座→自动替换→继续执行”的过程提示
- 最终执行结果中，餐厅步骤的目标从主选变更为备选（且能解释“为何替换”）
- 最终行程整体仍在时间窗口内（没有明显超窗）

## 观察点清单（评委最关心）

- 规划质量：是否满足人群/距离/时长窗口，时间分配是否自然  
- 执行质量：是否真的执行了 Mock 预约/排队动作，而不是只输出文本  
- 异常覆盖：无座场景是否触发补救；补救后是否继续把事情做完  
- 低打扰：系统能自己做的决策尽量自己做；只有越界才要求用户选择  

## 常见问题（仅写现象与处理）

- 现象：主服务报 “连接失败/超时”  
  - 处理：确认规划子服务也已启动（两个服务均需就绪）。
- 现象：首次启动很慢  
  - 处理：等待服务就绪后再请求；若项目有全局索引构建开关，请按项目说明避免重复构建导致阻塞。
