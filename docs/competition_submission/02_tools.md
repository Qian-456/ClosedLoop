# Tool 实现能力清单（含 Mock 行为）

本文件用于满足赛题交付物要求：提供完整的 Tool 实现代码（含 Mock API 调用）的能力说明。  
说明：为便于评委快速验收，本文件不贴具体代码路径，仅以“工具能力 + 输入输出 + Mock 行为 + 失败模式”进行结构化描述。

## Tool 总览

| Tool | 目的 | 关键输入（摘要） | 关键输出（摘要） | 常见失败模式 |
|---|---|---|---|---|
| 规划工具（Plan） | 从一句话目标生成 4–6 小时可执行行程方案 | 群体类型（family/friends）、预算、开始时间、总时长窗口、距离偏好、饮食/活动偏好、人数等 | 一套或多套结构化方案（含 steps、时间段、花费、通勤等） | 0 候选/无可行方案、规划服务不可用 |
| 调整工具（Adjust） | 用户对已生成方案做局部修改（替换/删除/调序等） | 目标方案标识、修改意图（换餐厅/换活动/更便宜/更近等） | 新的结构化方案（尽量保持整体不变） | 修改导致冲突/超窗、无可替换候选 |
| 执行工具（Execute） | 将确认后的方案转为可执行动作并执行（Mock） | plan_id、是否执行全部预约策略等 | 执行结果（逐步状态 + 最终总结 + 失败原因/替换信息） | 无座/无票、排队过久、库存扣减失败（Mock） |

## 1) 规划工具（Plan）

### 1.1 功能边界

- 输入：用户一句自然语言目标（或其结构化约束），覆盖亲子/朋友两类群体。
- 输出：4–6 小时窗口内的“综合方案”，至少包含：活动 → 正餐 → 后续活动（可选加购/可选步骤可作为附加项展示）。
- 不做：执行阶段的“硬性锁座/硬性锁票”。规划阶段可引入“可用性风险分”（软排序），但不以“是否有座”作为硬过滤，以保证 Pass@1 与稳定性。

### 1.2 输出数据的评委验收点

规划结果应具备以下信息（无需所有字段都展示给用户，但应存在于结构化产物中）：

- 行程 steps：每步的类型（活动/餐饮/通勤/可选项）、起止时间、预计耗时
- 人群适配理由（亲子友好/朋友聚会氛围等）
- 与 Mock 数据的一致性：餐饮 step 对应某个餐厅套餐（combo），活动 step 对应某个活动票种/套餐（package）

### 1.3 失败模式（规划侧）

- **0 候选/无可行方案**：用户约束过强导致召回/过滤后为空。系统需要输出“可解释的诊断信息”，引导用户放宽约束（预算、距离、时长、偏好等）。

## 2) 调整工具（Adjust）

### 2.1 功能边界

- 目标：在不推倒重来的前提下，对已生成方案做“局部替换/微调”。
- 常见修改：
  - 替换某个餐厅/活动（更近/更便宜/更适合带娃/更安静等）
  - 删除低优先级步骤
  - 调整顺序（例如“先吃再玩”）

### 2.2 冲突处理（与异常覆盖联动）

当用户修改导致时间冲突或超出窗口时，系统采用“分级修复”的思想（优先小改动）：

1) 吃 buffer（吸收 5–10 分钟级别的冲突）  
2) 压缩可压缩项目（有最小时长边界）  
3) 替换可替换项（不动用户锁定项，换更近/更短/低等待）  
4) 删除低优先级项（先删可选/加购/次要活动）  
5) 无法自动修复 → 输出结构化冲突报告，交给 Agent 解释并给 2–3 个按钮选项  

## 3) 执行工具（Execute）

### 3.1 功能边界

- 输入：用户确认后的方案标识（plan_id）以及“预约策略”（例如只预约第一段通勤/全部预约等）。
- 输出：执行过程状态（逐步）+ 最终结果摘要。
- 执行严格 Mock：不会真实下单；只在本地 Mock DB 内模拟扣减与排队。

### 3.2 Mock 行为（Side Effects）

执行过程中会对本地 Mock 数据产生可验证的变化（取决于实现策略，至少满足以下一项）：

- **库存扣减**：对 `reservations.json` 中对应 `target_id` 的 `capacity_remaining` 进行扣减或标记已占用
- **排队模拟**：当 `queue_required=true` 时，产生“取号/排队中/完成”的过程事件（并使用 `wait_minutes` 作为拟真等待）
- **执行事件**：对外输出“正在做什么/是否成功/失败原因/是否替换”的过程事件，便于前端或命令行展示

### 3.3 执行期异常自愈（主记忆点：无座自动替换）

当餐厅预订失败（无座/满员）时：

- 系统优先尝试“同槽位备选替换”（同区域/同预算区间/同人群适配/用餐类型相近），尽量保持后续行程不变。
- 仅当替换会显著改变体验（例如预算明显上浮、距离明显变远、人群适配下降）时，触发用户决策（Human-in-the-loop）。

## 4) 数据依赖关系（文件级）

- **restaurants.json**：餐厅与套餐（combo）数据源  
- **activities.json**：活动与票种/套餐（package）数据源  
- **reservations.json**：对 combo/package 在不同时间段的可用余量、排队与等待时间（capacity/queue/wait）  

验收建议：

- 规划输出中的餐饮/活动步骤应能在上述数据中找到对应条目  
- 执行后能观察到库存/排队相关字段的变化（或执行日志/事件的变化）
