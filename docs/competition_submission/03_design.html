设计文档（≤2页）：ClosedLoop 执行型本地生活 Agent（真实链路口径）

TL;DR

ClosedLoop 是一个执行型本地生活 Agent：用户一句话 → 生成 4–6 小时行程（玩→吃→后续）→ 用户确认 → Mock 执行预约/排队/库存扣减 → 遇到关键失败进入补齐（fixup）继续把事情做完。
设计目标不是“最炫”，而是响应快、稳定可复现、能落地（工具响应 ≤ 3 秒）。

背景（S）

本赛题要求在周末 4–6 小时窗口内，完成“规划 → 确认 → Mock 执行”的闭环，并满足速度约束（规划 ≤30 秒、单工具 ≤3 秒、端到端 ≤2 分钟）。

难点（C）

  • 规划侧：如果把“检索 + 组合规划 + 校验”都交给大模型，在真实场景（周边上百家店、用户偏好不明确）下容易出现：候选随机抽样、时间/预算计算不稳、需要多轮调用与工具补算导致等待变长，且难与平台用户画像/历史行为深度融合，结果往往退化为“热门但不个性化”的方案。
  • 执行侧：目标是在尽量少打扰用户的情况下完成 Mock 预约/下单/库存扣减等动作（本版本不做线上排队 Mock）；复杂点主要在“无座/无票/缺货”等失败处理与低打扰补齐（fixup），而非调用 API 本身。
  • 工程侧：工具与端到端的超时/降级/可恢复错误处理，决定了系统能否稳定演示与验收。

关键问题（Q）

如何在严格时延预算下，把“多约束规划 + Mock 执行 + 失败自愈”做成稳定、可复现、可快速验收的闭环？

方案（A）

  • 规划策略：用确定性模块兜住“可行性与稳定性”，LLM 负责“理解、解释与低打扰协商”；规划与重规划下沉到子服务以隔离负载。
  • 工具链路：以 plan_trip 作为单一入口承载“首次规划/改约束再规划”，通过 search_candidates → adjust_plan_item 支持“先搜再换”，执行侧用 execute_itinerary，失败进入 fixup_agent 低打扰补齐。
  • Mock 形态：Mock 数据字段对齐真实线上推荐/搜索可用的画像结构；对商家缺失字段，本版本采用离线规则/人工构造补齐，线上可用文本语义模型做抽取与估计作为演进方向。
  • 工程边界：本版本仅保留 Main(8000) / Plan(8001)（可选 Search 8002）三个相对“粗粒度”的服务边界（Plan 负责组合规划产出 itinerary + candidates；Search 负责在候选池内按关键词检索 top_k）；所有“下单/预约/排队”均为 Mock，且以内嵌模块实现，避免拆出更多微服务导致接口契约与排障成本上升。

真实架构（两服务 + 内部调用 + 可观测）

若 Markdown 渲染器不支持内嵌 SVG，可直接打开同目录的 03_design.html 查看离线版（图与样式都已内嵌）。

说明：

  • 两服务拆分用于模块化与可观测性：规划与再规划/组合搜索在 Plan(8001) 内完成，避免规划重负载影响主对话体验（Main 8000 负责对话路由与工具编排）。
  • 内部 HTTP 仅用于本地/容器内模块通信，所有“下单/预约/排队”均为 Mock。
  • Search(8002) 用于“用户先搜再换”的可选能力：当用户发起关键词搜索/替换时由 search_candidates 调用；不可用时主链路仍可退回再规划或提示换词重试。
  • 线上演进方向：可与平台既有能力融合（推荐侧用用户数据优化排序打分；搜索侧接入现有检索引擎），但不影响本次主链路稳定验收。

工具调用链路（用户动作闭环）

一句话需求
  ↓（plan_agent：语义理解+参数归一化）
plan_trip（≤3s）→ Main(8000) 内部 HTTP → Plan(8001) → 返回 itinerary + candidates
  ↓
用户确认 / 想改
  - 改约束/改结构：仍走 plan_trip（同一入口，体验一致）
  - 只想多看：generate_alternative_plans
  - 指定替换：search_candidates（调用 Search 8002；若超时/不可用则返回可恢复错误 → 提示用户换词重试或退回 plan_trip 再规划）→ adjust_plan_item
  ↓
确认执行
  ↓（execute_agent）
transfer_to_execute → execute_itinerary → mock_executor（库存/容量扣减+事件）
  ↓
若失败需要补齐
  ↓（fixup_agent）
候选1/2 或 search → adjust_and_execute_plan_item（替换+重试执行）

设计原则：

  • 确定性模块负责“可行性与稳定性”（时空约束、预算、回滚、一致性校验）；LLM 负责“理解、解释与低打扰协商”。
  • 规划阶段不把“是否有座/有票/有库存”等波动强的可用性做硬过滤：规划期生成备选并等待用户确认，执行期才做硬检查与兜底；同时避免过早锁定预约带来的资源挤占/恶意占座问题（对齐线上常见治理逻辑）。
  • 关键数据语义：规划期将 expected_wait_minutes（基于热度/评分的预计等待）计入耗时，并在步骤间默认预留 5 分钟 buffer；duration_std_dev 主要用于“用户修改后”的冲突修复（吸收小范围时间冲突）。执行期用 capacity_remaining / available_stock / stock 做硬检查与 Mock 扣减。

异常覆盖（≥3类，策略顺序）

1) 执行失败（无座/无票/库存不足）

  • 失败分层与低打扰推进：若该 step 未被用户指定/锁定（user_touched=false，适用于活动/餐厅/礼物），且备选不会导致超预算/超时或明显偏好受损（requires_confirmation=false），则静默替换并继续执行；否则进入 confirmation.status=needs_fixup，由 fixup_agent 给 1–2 个选项并继续推进（本版本主要支持 equivalent_only；strict/completion_first 暂未在全链路启用）。
  • 备选自动替换（等价优先）：执行期对 combo/package 做硬预订；失败时优先使用规划阶段为每个 step 预生成的 2 个 backup_candidates 进行替换（默认策略 replacement_policy=equivalent_only）。
  • 工程兜底：为保证可复现，执行前对运行时数据（runtime JSON）快照；当执行一致性校验失败（例如 Mock 扣减/Mock 支付相关一致性不满足）时，回滚运行时数据并返回可重试错误码。

2) 用户修改导致冲突（替换后超窗/超预算）

  • 替换走确定性修复：优先最小改动（吃 buffer/压缩可压缩项目），避免静默破坏体验。
  • 若替换后仍出现超窗/超预算，当前版本不会静默强行写入不可行方案，而是将该情况暴露为冲突场景，由 Agent 向用户解释需要重新调整。诸如“保留但延长”“撤销本次替换”“自动生成多种冲突解法”等能力，当前作为后续增强方向。

3) 无可行方案（0 候选 / 规划失败）

  • 当硬约束过强导致无法产出方案，系统返回可恢复错误，并引导用户放宽最小必要约束（预算、距离、窗口、偏好等）。
  • 当规划子服务不可用/超时（≤3s），主服务返回 TOOL_TIMEOUT，不中断对话，可重试或改用再规划。

取舍说明（补充）

  • 相比纯 LLM 端到端：本方案优先保证稳定验收与落地效率，避免多轮生成与工具补算导致等待变长、且难以稳定满足强时空约束。
  • 人机协同：采用轻量补齐（fixup_agent），只在关键失败点要求用户明确选择，其余尽量自动兜底，保证闭环成功率与演示稳定性。
