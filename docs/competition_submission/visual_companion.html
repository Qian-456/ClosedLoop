目标：在“真实可复现”的前提下，最大化命中评委关心点（规划质量 / 工具调用链路 / 异常覆盖 / 把事情做完）。

1) 评分点对齐：你要“主动展示”的四件事

  • 可执行闭环：一句话 → 规划 → 确认 → 执行（Mock side effects 可验证）。
  • 稳定性策略：规划期不把“可用性波动”当硬过滤，执行期才做硬检查 + 自动兜底。
  • 异常覆盖：至少 3 类（0 候选诊断 / 冲突修复 5 级 / 无座自动替换）。
  • 工程化：确定性模块做校验与修复，LLM 做理解/解释/协商（评委会更信）。
完整性 执行质量 / Pass@1 异常覆盖（可演示） 工程稳定性
建议答辩时口播关键词：deterministic validator、Plan 不硬查座、Execute 才硬扣库存、失败先自愈再问用户

2) 工具清单展示方式：从“描述能力”升级到“可验收证据”

  • 每个 Tool 不只写 I/O，要补 1 个“验收动作”（怎么稳定触发、如何观察结果）。
  • 执行工具必须展示 side effects：库存字段变化 / 排队事件 / 执行事件流。
  • 失败模式要“结构化”：失败原因 + fallback 次序 + 需要用户决策的边界。
Tool I/O Mock side effects Failure modes
关键点：把“我们实现了”变成“评委 30 秒内能看到变化/证据”。

3) 链路图：Planning → Confirm → Execute（评委最喜欢的那张图）

这张图的讲法：“我们把不稳定因素（可用性）后置到执行期，规划期只做风险排序；执行期失败不报错，先替换把事情做完。”

4) 异常覆盖三连：评委要的不是‘列举’，而是‘策略顺序’
