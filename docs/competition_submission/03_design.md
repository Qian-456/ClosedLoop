# 设计文档（≤2页）：ClosedLoop 执行型本地生活 Agent

## TL;DR

ClosedLoop 是一个执行型本地生活 Agent：用户一句话目标 → 生成 4–6 小时可执行行程（玩→吃→后续）→ 用户确认 → 自动执行预约/排队/通知（Mock）→ 遇到关键失败（如餐厅无座）可自动补救并继续把事情做完。

## 赛题约束摘要

- 场景：周末短时 4–6 小时；两类人群：亲子 / 朋友聚会  
- 要求：不仅推荐，还要“查座位/排队（Mock）+ 转为可执行动作（Mock）”  
- 性能：方案生成 ≤ 30 秒；工具响应 ≤ 3 秒；端到端 ≤ 2 分钟  
- 异常覆盖：至少 3 类（无座/无票/冲突/无可行方案等）

## 总体链路（Plan → Confirm → Execute）

```text
用户输入一句话
  ↓
约束抽取（群体、预算、距离、开始时间、时长窗口、偏好）
  ↓
候选召回（餐厅/活动/附加项）
  ↓
硬性过滤（时段/距离/人群等）
  ↓
重排打分（距离、人群适配、热度/体验等）
  ↓
行程规划（组合“玩→吃→后续”，计入通勤与时间弹性）
  ↓
用户确认（选择方案/允许自动替换边界）
  ↓
执行（Mock 预约/排队/通知）
  ↓
异常自愈（无座自动替换 / 冲突修复 / 0候选诊断）
```

设计原则：

- 规则/确定性模块负责“可行性与稳定性”；LLM 负责“理解、解释、协商与低打扰交互”。  
- 规划阶段避免把“高波动的可用性”做成硬过滤，以保证 Pass@1 与端到端稳定性。  
- 执行阶段对可用性做硬检查；失败时优先局部补救（而非把错误甩给用户）。

## Mock 数据契约（面向可执行性）

项目使用本地 Mock DB（无真实网络调用）模拟本地生活数据，保证可复现演示闭环：

- 餐厅数据：包含套餐（combo），每个 combo 有耗时、适用时段、是否需要预约  
- 活动数据：包含票种/套餐（package），每个 package 有开始时间/耗时/是否需要预约  
- 预约与排队：用 `reservations` 描述 combo/package 在各时段的余量与排队等待

关键语义：

- `duration_mins + duration_std_dev`：用于时间弹性，避免行程排得过满  
- `capacity_remaining`：执行期硬检查依据；为 0 表示无座/无票  
- `queue_required + wait_minutes`：拟真排队体验，可用于规划期风险提示与执行期过程展示

## 异常覆盖设计（≥3类）

### 1) 无座/无票（执行期）

目标：失败不报错、先补救，尽量把事情做完。

- 执行阶段对目标 combo/package 做硬预订  
- 若无座（capacity=0）：
  - 优先尝试同槽位备选替换（同区域/同预算/同人群适配/类型相近）  
  - 替换不越界则静默执行；越界则触发用户选择（Human-in-the-loop）  

### 2) 行程冲突（用户修改后）

采用“分级修复”理念，优先最小改动，保证体验与可解释性：

1) 吃 buffer 吸收小冲突  
2) 压缩可压缩项目（有最小时长边界）  
3) 替换可替换项（不动用户锁定项）  
4) 删除低优先级项（先删可选/加购/次要活动）  
5) 无法自动修复 → 输出结构化冲突报告 + 给 2–3 个可选项

### 3) 无可行方案（0 候选）

当预算/距离/时长等硬约束过强导致无法生成方案时：

- 系统输出可解释的诊断信息（例如“预算过低/距离过近/窗口过短”导致过滤为空）  
- 引导用户放宽最小必要约束，而不是直接失败结束

## 为什么不是纯 LLM 端到端规划

多约束、强时空、强可执行任务（距离/时间窗/库存/排队）对稳定性要求高。纯 LLM 端到端容易出现“看起来合理但不可执行”的方案。  
因此系统采用“确定性规划骨架 + LLM 在理解与交互上增益”的路线：既保证可执行与可复现，也能体现 Agent 的低打扰与自愈能力。

## 验证方式（与 Demo 对齐）

通过可控的 Mock 数据（reservations 中设定少数时段 capacity=0）稳定触发“无座→自动替换”，并在执行过程与最终结果中观察：

- 是否完成“规划→确认→执行”闭环  
- 是否在失败后自动补救并继续完成  
- 是否保持行程窗口与整体体验的稳定性
